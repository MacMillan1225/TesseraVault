# TesseraScript 新组件开发规范

本文档用于约束未来在 `TesseraScript` 中新增组件时的目录结构、代码风格、注册方式、测试方式与完整工作流。

## 1. 开发目标

新组件必须服务于以下目标：

- 运行在 Obsidian + DataviewJS 环境
- 不依赖插件开发模式
- 不新增全局变量，只使用 `globalThis.Tessera`
- 组件通过 `Tessera.define(...)` 注册
- 用户优先通过 `Tessera.use(...)` 使用组件
- 模块作者保持接近 CommonJS 的写法

## 2. 标准目录结构

每个新组件都应放在 `TesseraScript/components/<name>/` 下，并尽量包含以下文件：

```text
TesseraScript/
  components/
    <name>/
      index.js
      style.css
      config.json
      README.md
      demo.md
```

说明：

- `index.js`：组件注册入口，必须存在
- `style.css`：组件样式，建议存在
- `config.json`：默认配置，可选但推荐
- `README.md`：组件说明，推荐存在
- `demo.md`：示例笔记，推荐存在

## 3. 命名规范

### 3.1 组件目录名

目录名使用小写英文，必要时使用短横线，例如：

- `progress`
- `card`
- `heatmap`
- `task-list`

### 3.2 模块 ID

组件注册时统一使用：

```js
components/<name>
```

例如：

```js
Tessera.define("components/task-list", function (require, module, exports) {
  // ...
});
```

不要使用：

- 反斜杠
- 文件扩展名
- 大小写混合路径

### 3.3 CSS 类名前缀

每个组件必须使用自己独立的前缀，避免污染其他笔记样式。

例如 `task-list` 组件应该使用：

- `ts-task-list`
- `ts-task-list__header`
- `ts-task-list__item`

不要写无前缀类名，例如：

- `.header`
- `.item`
- `.wrapper`

## 4. `index.js` 编写规范

### 4.1 基本模板

所有组件入口都必须按下面结构书写：

```js
Tessera.define("components/<name>", function (require, module, exports) {
  const dom = require("../core/dom");

  function <name>(options = {}) {
    return dom.createElement("div", {
      className: "ts-<name>",
      text: "demo",
    });
  }

  module.exports = <name>;
  module.exports.<name> = <name>;
});
```

### 4.2 相对依赖要求

组件内部引用其他模块时，必须使用 `require()`，并尽量使用相对路径。

例如：

```js
const dom = require("../core/dom");
```

不要在组件里直接写浏览器全局耦合过重的逻辑，除非组件本身确实在生成 DOM。

### 4.3 导出要求

单函数组件统一使用：

```js
module.exports = component;
module.exports.component = component;
```

如果组件名是 `progress`，则写法为：

```js
module.exports = progress;
module.exports.progress = progress;
```

这样做的目的是：

- 支持 `const progress = require("components/progress")`
- 支持聚合入口里用 `progress.progress || progress`

## 5. 样式文件规范

### 5.1 `style.css` 必须只服务当前组件

组件样式只能作用于自己的前缀范围内。

例如：

```css
.ts-task-list {}
.ts-task-list__item {}
.ts-task-list__item.is-done {}
```

不要直接覆盖：

```css
div {}
ul {}
.card {}
```

### 5.2 样式要求

最少要保证：

- 组件结构能看清
- 间距合理
- 在桌面端与移动端不明显错位
- 尽量兼容 Obsidian 主题变量，如 `var(--text-normal)`

## 6. 配置文件规范

如果组件有稳定的默认参数，建议放入 `config.json`。

推荐统一通过 `TesseraScript/core/config.js` 读取与合并配置，不要每个组件自己手写一套 JSON 读取逻辑。

适合放到 JSON 的内容：

- 默认尺寸
- 默认颜色等级
- 默认间距
- 默认标题文案
- 默认布尔开关，例如是否显示某一段内容
- 简单数组或数字阈值

不适合放到 JSON 的内容：

- DOM 节点实例
- 函数
- 强依赖运行时上下文的对象

### 6.1 推荐的配置分层

组件默认配置推荐分成三层：

1. `index.js` 中的 fallback 默认值
2. 组件目录下 `config.json` 中的结构化默认值
3. 用户调用组件时传入的 `options`

最终渲染配置应满足：

```text
fallback + config.json + options = resolvedConfig
```

### 6.2 推荐的接入方式

组件里推荐使用：

```js
const createConfigController = require("../core/config");
const config = createConfigController();

const defaultTaskListConfig = {
  flags: {
    showHeader: true,
  },
  layout: {
    gap: "12px",
  },
};

const taskListConfig = config.createScope({
  path: "TesseraScript/components/task-list/config.json",
  fallback: defaultTaskListConfig,
});
```

组件渲染时：

```js
const resolved = taskListConfig.merge(options);
```

### 6.3 路径要求

`config.createScope({ path })` 中的 `path` 当前必须写 vault 绝对路径风格，例如：

```js
path: "TesseraScript/components/task-list/config.json"
```

不要写：

```js
path: "./config.json"
```

原因是 `core/config.js` 当前走的是文件读取路径，不是 Tessera 模块相对路径解析。

### 6.4 是否导出 `loadConfig`

如果组件需要让调用方控制“首次渲染前是否必须先完成 JSON 默认配置加载”，则推荐导出：

```js
module.exports.loadConfig = loadTaskListConfig;
```

约定如下：

- 组件函数内部可以触发配置预热，但这不代表会等待加载完成
- 如果调用方要求“第一次渲染就严格使用 `config.json`”，则应先：`await component.loadConfig()`
- 如果 fallback 已足够合理，也可以直接渲染，不强制等待

### 6.5 推荐的 JSON 结构

推荐按语义分组，不要把所有字段都平铺在第一层：

- `flags`
- `layout`
- `colors`
- `text`

数组默认按整体替换处理，不做逐项深合并。

## 7. 聚合入口规范

如果新组件属于“高频组件”，则必须同步更新 `TesseraScript/index.js`，把它加入聚合入口。

模板：

```js
const taskList = require("components/task-list");

module.exports = {
  // ...existing exports
  taskList: taskList.taskList || taskList,
};
```

## 8. 别名规范

如果新组件希望用户用短名直接调用，则必须更新 `TesseraScript/tessera.bootstrap.js` 中的默认别名：

```js
Tessera.alias({
  "task-list": "components/task-list",
});
```

注意：

- 注册 alias 本身不会加载模块
- 只有真正执行 `Tessera.use("task-list")` 时，才会去查找模块是否存在

## 9. README 与 demo 要求

### 9.1 `README.md`

至少写清楚：

- 组件用途
- 参数说明
- 最小示例
- 返回值类型
- 如果有 `config.json`，必须说明默认配置文件路径
- 如果导出了 `loadConfig()`，必须说明它是否是“首次渲染前必需步骤”

### 9.2 `demo.md`

至少给一段可直接复制的 `dataviewjs` 示例。

## 10. 新组件完整工作流

下面是未来新增组件时必须执行的完整流程。

### 步骤 1：明确组件职责

先回答这几个问题：

- 这是基础组件还是组合视图？
- 它应该放到 `components/` 还是 `views/`？
- 它是否应该被 `Tessera.use("components")` 暴露？
- 它是否需要默认短名 alias？

如果它只是基础可复用渲染单元，放 `components/`。
如果它是多个组件拼出来的完整页面片段，放 `views/`。

### 步骤 2：创建目录和文件

创建：

- `TesseraScript/components/<name>/index.js`
- `TesseraScript/components/<name>/style.css`
- `TesseraScript/components/<name>/config.json`
- `TesseraScript/components/<name>/README.md`
- `TesseraScript/components/<name>/demo.md`

### 步骤 3：先写最小可运行组件

优先写一个最小版本，保证：

- 可以注册
- 可以 `Tessera.use()` 到
- 可以返回 DOM

先不要过早做复杂逻辑。

如果该组件存在稳定默认配置，应同时决定：

- 是否需要 `config.json`
- fallback 默认值放在哪
- 是否需要导出 `loadConfig()` 让外部等待首次配置加载完成

### 步骤 4：补样式

为最小 DOM 结构补 `style.css`，至少保证：

- 结构层次清楚
- 类名前缀正确
- 不污染全局

### 步骤 5：决定是否加入聚合入口

如果是高频组件：

- 更新 `TesseraScript/index.js`
- 让用户可通过 `Tessera.use("components")` 拿到它

如果不是高频组件，可以只保留独立模块入口。

### 步骤 6：决定是否加入短名 alias

如果希望用户直接写：

```js
const taskList = Tessera.use("task-list");
```

则更新 `TesseraScript/tessera.bootstrap.js` 中的 alias。

### 步骤 7：写 demo

必须写最小可复制示例，至少验证：

- 脚本加载顺序正确
- `Tessera.use()` 可用
- 组件确实能 append 到 `dv.container`

### 步骤 8：本地检查

每次新增或修改组件后，至少做下面检查：

1. `index.js` 语法是否正确
2. 模块 ID 是否正确
3. 相对 `require()` 是否正确
4. alias 是否和模块 ID 一致
5. 聚合入口是否正确导出
6. `demo.md` 是否可复制运行
7. 若使用 `config.json`，路径是否写成 vault 绝对路径
8. 若依赖首次配置加载，示例里是否显式 `await component.loadConfig()`

### 步骤 9：Obsidian 手工验证

在实际笔记里验证：

1. 先加载 `tessera.bootstrap`
2. 再加载依赖 core 模块
3. 再加载组件 `index.js`
4. 如果用了聚合入口，再加载 `TesseraScript/index`
5. 调用 `Tessera.use(...)`
6. 把返回的 DOM append 到页面

### 步骤 10：补文档

最后更新这些文档中的相关部分：

- 组件目录下 `README.md`
- 组件目录下 `demo.md`
- 如有必要，更新 `TesseraScript 使用示例.md`
- 如有必要，更新本规范

## 11. 推荐验收清单

一个新组件只有在下面项目都满足时，才算完成：

- 已存在组件目录
- 已存在 `index.js`
- 已使用 `Tessera.define(...)`
- 已正确导出 `module.exports`
- 已存在最小 `style.css`
- 已能在 DataviewJS 中渲染
- 如需高频暴露，已加入 `index.js`
- 如需短名调用，已加入 alias
- 已有 `README.md`
- 已有 `demo.md`

## 12. 一个新组件模板

```js
Tessera.define("components/example", function (require, module, exports) {
  const dom = require("../core/dom");
  const createCSSController = require("../core/css");
  const createConfigController = require("../core/config");
  const css = createCSSController();
  const config = createConfigController();
  let stylePromise = null;
  const defaultExampleConfig = {
    title: "Example",
  };
  const exampleConfig = config.createScope({
    path: "TesseraScript/components/example/config.json",
    fallback: defaultExampleConfig,
  });

  function ensureStyles() {
    if (!stylePromise) {
      stylePromise = css.ensure({
        id: "components-example",
        path: "TesseraScript/components/example/style.css",
      }).catch((error) => {
        stylePromise = null;
        console.warn("[Tessera] Failed to load example styles.", error);
      });
    }

    return stylePromise;
  }

  function loadExampleConfig(options = {}) {
    return exampleConfig.load(options).catch(() => exampleConfig.get());
  }

  function example(options = {}) {
    ensureStyles();
    const resolved = exampleConfig.merge(options);

    return dom.createElement("section", {
      className: "ts-example",
      children: [
        dom.createElement("div", {
          className: "ts-example__title",
          text: resolved.title,
        }),
      ],
    });
  }

  module.exports = example;
  module.exports.example = example;
  module.exports.loadConfig = loadExampleConfig;
});
```

对应最小样式：

```css
.ts-example {
  padding: 12px;
  border-radius: 12px;
  background: rgba(120, 140, 160, 0.12);
}

.ts-example__title {
  font-size: 14px;
  font-weight: 600;
}
```

## 13. 自动样式注入固定模板

未来所有需要自带样式的组件，都应默认采用下面这段结构。

```js
const createCSSController = require("../core/css");
const css = createCSSController();
let stylePromise = null;

function ensureStyles() {
  if (!stylePromise) {
    stylePromise = css.ensure({
      id: "components-example",
      path: "TesseraScript/components/example/style.css",
    }).catch((error) => {
      stylePromise = null;
      console.warn("[Tessera] Failed to load example styles.", error);
    });
  }

  return stylePromise;
}
```

组件函数开头必须调用：

```js
function example(options = {}) {
  ensureStyles();
  // render...
}
```

这样做的原因：

- 第一次渲染时自动注入样式
- 后续重复渲染时不会重复插入
- 如果样式加载失败，下次仍可重试

固定要求：

- `id` 必须和组件名保持稳定对应，例如 `components-example`
- `path` 必须指向该组件目录下的 `style.css`
- 不要把样式注入逻辑散落到多个函数里
- 不要要求用户手动复制 CSS 文本

## 14. 关于失败行为的规则

如果你给一个组件注册了 alias，但并没有真正注册对应模块：

- `Tessera.alias({ foo: "components/foo" })` 不会报错
- `Tessera.use("foo")` 会在运行时报错
- 报错应类似：`[Tessera] Module not found: components/foo`

因此规范要求：

- 先写模块
- 再决定是否加 alias
- 加 alias 后要立刻做一次实际调用验证
