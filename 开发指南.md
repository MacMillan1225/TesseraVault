# TesseraScript 开发指南

本指南用于统一 `开发详情.md`、`规范.md`、根目录 `README.md` 与 `TesseraScript/components/example/README.md` 中的开发约束、目录说明、运行时规则与新组件开发流程。

如果这些文档之间存在冲突，优先级按以下顺序处理：

1. `TesseraScript/components/example/README.md`
2. 当前仓库中的实际实现，尤其是 `TesseraScript/components/card/`
3. `规范.md`
4. 根目录 `README.md`
5. `开发详情.md`

也就是说，本指南不是简单拼接旧文档，而是以当前代码可执行、可维护、可复制为目标做出的统一版本。

## 1. 项目定位

`TesseraScript` 是一套运行在 `Obsidian + DataviewJS` 环境中的轻量脚本组件库。

它的目标不是做传统前端工程，而是做一套：

1. 可以直接通过 `dataviewjs` 调用
2. 不依赖 Obsidian 插件开发模式
3. 不依赖 npm、打包器、TypeScript 构建链
4. 支持模块化拆分
5. 支持 JS / CSS / JSON 配置分离
6. 让用户优先通过短名或聚合入口使用组件

## 2. 总体原则

### 2.1 单一全局命名空间

全局只允许暴露一个对象：

```js
globalThis.Tessera
```

不要额外挂载多个全局变量。

错误示例：

```js
window.card = ...
window.progress = ...
window.fileUtils = ...
```

正确思路是：

1. 所有模块通过 `Tessera.define(...)` 注册
2. 所有模块通过 `Tessera.require(...)` 或 `Tessera.use(...)` 访问

### 2.2 用户导入体验优先

对最终使用者而言，首选调用形式应该尽量短。

推荐：

```js
const card = Tessera.use("card");
const { progress, heatmap } = Tessera.use("components");
const ui = Tessera.use("@ui");
```

可以支持但不推荐让用户长期写底层路径：

```js
const card = Tessera.require("components/card");
```

### 2.3 模块作者保持 CommonJS 风格

模块内部采用接近 CommonJS 的写法：

```js
Tessera.define("components/card", function (require, module, exports) {
  const dom = require("../core/dom");

  function card(options = {}) {
    return dom.createElement("div");
  }

  module.exports = card;
  module.exports.card = card;
});
```

支持：

1. `module.exports = ...`
2. `exports.xxx = ...`
3. `require("./relative")`
4. `require("../relative")`

这里的 `require` 是 Tessera 运行时提供的，不是 Node 的原生 `require`。

### 2.4 先注册，再使用

DataviewJS 环境下的使用顺序是：

1. 先加载 `tessera.bootstrap.js`
2. 再加载所需 core 模块
3. 再加载组件模块
4. 如有需要，再加载 `TesseraScript/index.js`
5. 最后通过 `Tessera.use(...)` 或 `Tessera.require(...)` 获取模块并渲染

这也是为什么当前模块系统可以保持同步 `require()` 体验。

## 3. 当前目录结构与职责

```text
TesseraScript/
  components/
    card/
    heatmap/
    progress/
    example/
  core/
    config.js
    css.js
    dom.js
    file.js
    style.js
  shared/
    base.css
    theme.css
  views/
```

### 3.1 `TesseraScript/core/`

当前核心能力主要来自以下模块：

1. `core/file.js`
   负责通过 Obsidian `app.vault` 读取 vault 中的文件内容
2. `core/css.js`
   负责读取并注入 CSS，支持 `ensure`、去重、更新、删除
3. `core/dom.js`
   负责创建 DOM、挂 class、挂 style、挂 attrs、追加 children
4. `core/config.js`
   负责读取 JSON 配置、管理 fallback、做深度合并、生成 scope
5. `core/style.js`
   当前存在，但不是新组件开发主流程的核心依赖

核心层只负责通用能力，不负责具体组件渲染。

### 3.2 `TesseraScript/components/`

这是用户直接使用的组件层。

组件的核心要求是：

1. 由 `Tessera.define("components/<name>", ...)` 注册
2. 最终直接返回 HTML 元素
3. 能直接 append 到 DataviewJS 容器

例如：

```js
dv.container.appendChild(card(options));
```

### 3.3 `TesseraScript/shared/`

目前 `shared/base.css` 和 `shared/theme.css` 还是空的，还没有形成稳定依赖。

因此当前开发新组件时：

1. 不要把它们当成必须依赖
2. 组件应尽量在自身目录内自给自足
3. 如果要使用 Obsidian 的主题变量，应直接使用 `var(--text-normal)`、`var(--background-secondary)` 这类变量

### 3.4 `TesseraScript/views/`

当前还没有形成完整视图层规范。

临时原则：

1. 可复用单元放 `components/`
2. 多组件组合出的完整页面片段以后再放 `views/`

如果一个东西本质上是单个可复用渲染单元，就先放 `components/`。

## 4. 运行时与入口文件

### 4.1 `TesseraScript/tessera.bootstrap.js`

这个文件负责建立整个运行时。

必须理解它的职责：

1. 初始化 `globalThis.Tessera`
2. 维护 `modules` 注册表
3. 维护 `cache` 模块缓存
4. 维护 `aliases` 别名映射
5. 提供 `define`、`register`、`require`、`use`、`resolve`、`alias`、`has`
6. 处理相对路径解析
7. 注册默认 alias

当前推荐理解的对外 API：

```js
globalThis.Tessera = {
  version,
  define,
  register,
  require,
  use,
  resolve,
  alias,
  has,
  modules,
  cache,
  aliases,
}
```

### 4.2 `TesseraScript/index.js`

这是聚合入口。

它的职责是把高频组件集中导出，让用户可以这样使用：

```js
const { card, progress, heatmap } = Tessera.use("components");
```

所以一个组件是否要加入 `index.js`，取决于它是否属于高频基础组件。

### 4.3 默认 alias 设计

当前运行时的设计目标是让这些调用优先成立：

```js
Tessera.use("progress")
Tessera.use("card")
Tessera.use("heatmap")
Tessera.use("components")
Tessera.use("@ui")
```

新组件如果希望拥有短名调用体验，就要在 `tessera.bootstrap.js` 中注册 alias。

## 5. 用户使用方式

推荐的 DataviewJS 加载顺序：

```dataviewjs
await dv.view("TesseraScript/tessera.bootstrap");
await dv.view("TesseraScript/core/dom");
await dv.view("TesseraScript/core/file");
await dv.view("TesseraScript/core/css");
await dv.view("TesseraScript/core/config");
await dv.view("TesseraScript/components/card/index");
```

然后：

```js
const card = Tessera.use("card");
dv.container.appendChild(card({ title: "Hello" }));
```

也可以在加载 `TesseraScript/index.js` 后，通过聚合入口使用：

```js
const { card, heatmap } = Tessera.use("components");
```

## 6. 新组件的标准目录结构

每个新组件都应放在：

```text
TesseraScript/components/<name>/
```

标准目录建议完整包含：

```text
TesseraScript/components/<name>/
  index.js
  style.css
  config.json
  README.md
  demo.md
```

说明：

1. `index.js`
   组件入口，必须存在
2. `style.css`
   组件公共样式，强烈建议存在
3. `config.json`
   组件默认配置，强烈建议存在
4. `README.md`
   组件 API 和使用说明，必须存在
5. `demo.md`
   可复制的演示与验证文件，必须存在

当前仓库已将 `TesseraScript/components/example/` 升级为标准模板，新组件开发时应优先复制它，而不是从零开始写。

## 7. 命名规范

### 7.1 组件目录名

目录名使用小写英文，必要时用短横线：

1. `card`
2. `heatmap`
3. `task-list`
4. `stat-card`

### 7.2 模块 ID

组件注册时统一使用：

```js
components/<name>
```

例如：

```js
Tessera.define("components/stat-card", function (require, module, exports) {
  // ...
});
```

不要使用：

1. 反斜杠
2. 文件扩展名
3. 混乱大小写

### 7.3 CSS 类名前缀

每个组件必须使用自己独立的类名前缀。

例如 `stat-card`：

```css
.ts-stat-card {}
.ts-stat-card__header {}
.ts-stat-card__value {}
```

不要写：

```css
.header {}
.item {}
.wrapper {}
```

### 7.4 CSS 变量前缀

组件自己的 CSS 变量也必须带组件前缀。

例如：

```css
--ts-stat-card-padding
--ts-stat-card-radius
--ts-stat-card-background-light
```

这个规则以 `components/card` 的当前实现为准。

## 8. 一个完整组件应该具备什么

参考 `components/card` 和当前 `components/example`，一个完整组件至少应具备这些能力：

1. 通过 `Tessera.define(...)` 注册
2. 通过 `require("../core/dom")` 使用 DOM 工具
3. 通过 `require("../core/css")` 加载组件 CSS
4. 通过 `require("../core/config")` 加载组件默认配置
5. 直接返回 HTML 元素
6. 尽量暴露关键子节点给调用方访问
7. 支持日间和夜间模式
8. 支持单实例样式覆盖
9. 支持 `config.json` 全局默认配置 + 运行时局部覆盖
10. 附带 `README.md` 与 `demo.md`

## 9. 新组件开发流程

以下流程是当前仓库推荐的正式开发流程。

### 步骤 1：先判断它是不是组件

先回答：

1. 这是基础渲染单元，还是多个组件拼起来的完整视图？
2. 它是否会被多个笔记或页面复用？
3. 它是否应该单独返回一个 DOM 元素？

如果答案是“它是基础可复用渲染单元”，就放到 `components/`。

### 步骤 2：复制模板目录

复制：

```text
TesseraScript/components/example/
```

到：

```text
TesseraScript/components/<name>/
```

然后改掉模板中的所有 example 标识。

### 步骤 3：修改 `index.js` 中的标识和路径

至少要替换这些内容：

1. 模块 id
   `components/example` -> `components/<name>`
2. CSS 注入 id
   `components-example` -> `components-<name>`
3. `style.css` 路径
4. `config.json` 路径
5. 导出函数名
6. CSS 类名前缀

如果这些名字漏改，复制第二个组件时会直接冲突。

### 步骤 4：先定义默认配置结构

在正式写 DOM 之前，先定义组件对外开放哪些配置。

推荐至少分成这几类：

1. 基础数据字段
   比如 `title`、`meta`、`value`、`text`、`items`
2. 显示开关
   比如 `flags.showHeader`
3. 布局参数
   比如 `layout.padding`、`layout.radius`
4. 颜色参数
   比如 `colors.light` / `colors.dark`
5. 单实例样式入口
   比如 `styles.root`、`styles.title`

这些默认值需要写在两处：

1. `index.js` 里的 fallback 对象
2. `config.json`

这样即使配置文件没有加载成功，组件也仍然可用。

### 步骤 5：用 `core/config` 创建配置作用域

标准写法参考当前模板：

```js
const createConfigController = require("../core/config");
const config = createConfigController();

const widgetConfig = config.createScope({
  path: "TesseraScript/components/your-component/config.json",
  fallback: defaultWidgetConfig,
});
```

建议同时导出：

```js
module.exports.loadConfig = loadWidgetConfig;
module.exports.getDefaultConfig = widgetConfig.get;
```

这样：

1. 用户可以预加载默认配置
2. 开发者可以查看最终 fallback 结构

### 步骤 6：用 `core/css` 统一管理样式注入

组件不要要求用户手动复制 CSS。

固定模式是：

```js
const createCSSController = require("../core/css");
const css = createCSSController();
let stylePromise = null;

function ensureStyles() {
  if (!stylePromise) {
    stylePromise = css
      .ensure({
        id: "components-your-component",
        path: "TesseraScript/components/your-component/style.css",
      })
      .catch((error) => {
        stylePromise = null;
        console.warn("[Tessera] Failed to load your-component styles.", error);
      });
  }

  return stylePromise;
}
```

组件函数开头应调用：

```js
ensureStyles();
```

这样做的作用：

1. 第一次渲染时自动加载样式
2. 后续重复渲染不重复插入
3. 加载失败后下次还能重试

### 步骤 7：用 `core/dom` 创建结构

不要散落大量原生 DOM 样板代码。

优先使用：

```js
dom.createElement(tagName, {
  className,
  attrs,
  style,
  text,
  html,
  children,
})
```

原因：

1. 与当前仓库风格一致
2. 结构更紧凑
3. 更适合后续维护和复制

### 步骤 8：组件必须直接返回 HTML 元素

最终主导出必须是一个返回 DOM 的函数。

例如：

```js
function statCard(options = {}) {
  return dom.createElement("article", { ... });
}

module.exports = statCard;
module.exports.statCard = statCard;
```

这样用户才能直接：

```js
dv.container.appendChild(statCard({...}));
```

### 步骤 9：暴露关键子节点

如果组件内部有多个结构块，建议把关键节点挂到根节点上。

例如：

```js
const root = dom.createElement("article", ...);

root.parts = {
  header,
  title,
  body,
  footer,
};

return root;
```

这样调用方可以继续做轻量定制：

```js
const node = statCard({...});
node.parts.title.textContent = "Updated";
```

当前模板已经内置了 `root.parts` 这种模式，建议沿用。

### 步骤 10：区分公共样式与单实例样式

这是当前组件设计里非常重要的一点。

要区分两个层次：

1. 公共结构样式写进 `style.css`
2. 同页面不同实例之间可能变化的样式，通过根节点内联 style 或 CSS 变量传入

例如：

1. `display`、布局方向、字号层级、默认交互，适合写在 `style.css`
2. `padding`、`radius`、`background`、`border`、局部颜色，适合写成根节点变量或 `styles.*`

这条规则在 `components/card` 里非常明显：

1. 公共样式在 `style.css`
2. 单卡配置通过 `--ts-card-*` 变量和 `styles.*` 内联传入

### 步骤 11：兼容 Obsidian 日间和夜间模式

Obsidian 会在 `body` 上挂：

1. `theme-light`
2. `theme-dark`

新组件必须兼容这两个类。

推荐方式：

1. 默认配置里提供 `colors.light` 和 `colors.dark`
2. CSS 中根据 `body.theme-light` / `body.theme-dark` 切换当前变量

模式参考 `components/card`：

```js
colors: {
  light: { ... },
  dark: { ... },
}
```

```css
body.theme-light .ts-your-component {
  --ts-your-component-background-current: var(--ts-your-component-background-light);
}

body.theme-dark .ts-your-component {
  --ts-your-component-background-current: var(--ts-your-component-background-dark);
}
```

同时推荐保留共享颜色覆盖能力，即：

```js
colors.background
colors.border
```

这种扁平字段可以同时覆盖 light/dark 两套颜色，方便在单实例里动态算色值。

### 步骤 12：正确处理默认值合并顺序

组件内部推荐的默认值来源顺序是：

1. 代码里的 fallback 默认值
2. `config.json` 中的默认配置
3. 运行时传入的 `options`

典型写法：

```js
const resolved = widgetConfig.merge(options);
```

这能保证：

1. 页面级风格可以统一写入 `config.json`
2. 单次调用还能再局部覆盖

### 步骤 13：决定是否加入聚合入口

如果这是高频基础组件，应更新：

```text
TesseraScript/index.js
```

模式：

```js
const statCard = require("components/stat-card");

module.exports = {
  ...,
  statCard: statCard.statCard || statCard,
};
```

如果它不是高频组件，可以只保留独立入口，不一定必须加入聚合导出。

### 步骤 14：决定是否加入短名 alias

如果希望用户可以直接：

```js
const statCard = Tessera.use("stat-card");
```

就需要更新：

```text
TesseraScript/tessera.bootstrap.js
```

模式：

```js
alias({
  "stat-card": "components/stat-card",
});
```

注意：

1. alias 注册本身不会检查模块是否已存在
2. 真正执行 `Tessera.use("stat-card")` 时才会解析并加载模块
3. 如果 alias 指向不存在的模块，会在运行时抛出 `Module not found`

### 步骤 15：写 `demo.md`

每个组件都必须提供完整示例，而不是只放一段极简代码。

建议至少包含：

1. 标准加载顺序
2. 最小可运行示例
3. 常见配置示例
4. 单实例样式覆盖示例
5. 日间/夜间主题示例
6. 多实例并排或组合示例
7. 如果有 `parts` 暴露，演示如何访问子节点

`demo.md` 不只是展示功能，也用于开发回归验证。

### 步骤 16：写 `README.md`

组件目录下的 `README.md` 应作为开发者和使用者都能直接参考的 API 文档。

建议至少写清楚：

1. 组件用途
2. 适用场景
3. 核心特点
4. 标准加载方式
5. 基础字段说明
6. `flags` 说明
7. `layout` 说明
8. `colors` 说明
9. `styles` 说明
10. 返回值说明
11. 推荐用法
12. 不推荐的用法
13. 配置加载方式

`components/card/README.md` 可以作为当前最成熟的参考范例。

## 10. 代码风格与实现约束

### 10.1 组件应优先自包含

每个组件应尽量把：

1. DOM 结构
2. 样式加载
3. 默认配置
4. 文档与 demo

都收敛在自己的目录内。

### 10.2 不要过度工程化

当前明确不需要引入：

1. 打包器
2. npm 包管理依赖
3. TypeScript 构建链
4. 插件设置页
5. 异步模块加载器
6. 热更新系统
7. 复杂 class-heavy 设计

优先目标是：

1. 简单
2. 可读
3. 可复制
4. 能直接在 Obsidian DataviewJS 中运行

### 10.3 不要污染全局样式

组件 CSS 只能作用于自己的命名空间。

不要写：

```css
div {}
ul {}
.card {}
```

要写：

```css
.ts-card {}
.ts-card__body {}
```

### 10.4 JSON 配置中放什么，不放什么

适合放进 `config.json`：

1. 默认尺寸
2. 默认文案
3. 默认间距
4. 默认颜色
5. 默认布局参数

不适合放进 `config.json`：

1. DOM 节点实例
2. 函数
3. 强依赖运行时上下文的对象

### 10.5 错误处理

样式加载失败、配置加载失败时，不要让整个组件直接崩掉。

推荐做法：

1. `console.warn(...)`
2. 回退到当前内置默认值

这也是当前 `components/card` 和 `components/example` 的实现方式。

## 11. 最小开发模板

新组件请优先复制 `TesseraScript/components/example/`。

它当前已经内置：

1. `ensureStyles()` 样式只注入一次
2. `loadConfig()` 配置加载
3. `config.merge(options)` 默认值合并
4. `colors.light` / `colors.dark` 双主题支持
5. 扁平颜色字段兼容
6. `root.parts` 子节点暴露
7. `styles.*` 单实例内联样式入口

## 12. card 组件提供的现实参考

如果你不确定抽象规则该怎么落地，请优先参考 `TesseraScript/components/card/`，因为它当前是仓库里最完整的标准组件之一。

它体现了以下实际开发模式：

1. `defaultCardConfig` 作为 fallback
2. `config.createScope(...)` 管理配置
3. `css.ensure(...)` 注入样式
4. `mergeStyles(...)` 合并内联样式
5. `colors.light` / `colors.dark` 双主题颜色
6. 共享颜色字段兼容
7. 把变化频繁的样式转成 `--ts-card-*` CSS 变量
8. 允许 `content` 与 `children` 两种内容入口
9. 用 `README.md` 清晰描述 API
10. 用 `demo.md` 展示实际调用方式

如果一个新组件的设计与你不确定是否合理，可以先对照 card 组件，看它是否遵守了同样的层次和接口设计。

## 13. 验证与验收清单

一个新组件在完成前，至少要通过下面这些检查。

### 13.1 静态检查

1. `index.js` 语法正确
2. 模块 id 正确
3. 所有 `require()` 路径正确
4. `config.json` 是合法 JSON
5. CSS 类名前缀正确
6. CSS 变量前缀正确

### 13.2 入口检查

1. 如果加入聚合入口，`TesseraScript/index.js` 已正确导出
2. 如果加入短名 alias，`TesseraScript/tessera.bootstrap.js` 已正确注册
3. alias 与模块 id 完全一致

### 13.3 文档检查

1. `README.md` 已更新
2. `demo.md` 已更新
3. 示例中的字段名与实际 API 一致
4. 示例中的加载顺序可运行

### 13.4 Obsidian 手工验证

至少在真实笔记里验证：

1. 能成功加载 `tessera.bootstrap`
2. 能成功加载所需 core 模块
3. 能成功加载组件 `index.js`
4. `Tessera.use(...)` 可以拿到组件
5. 返回的 DOM 能 append 到 `dv.container`
6. light 模式下显示正常
7. dark 模式下显示正常
8. 同页多个实例时，单实例样式不会互相污染

## 14. 推荐的开发顺序

如果从零做一个新组件，推荐按这个顺序推进：

1. 判断它属于 `components/` 还是 `views/`
2. 复制 `components/example/`
3. 改模块名、路径、前缀
4. 设计默认配置结构
5. 先写最小可运行 DOM
6. 接入 `config`、`css`、主题变量
7. 补 `style.css`
8. 增加单实例覆盖入口
9. 视需要暴露 `parts`
10. 补 `README.md`
11. 补 `demo.md`
12. 视需要注册 alias 与聚合导出
13. 做静态与手工验证

## 15. 最终目标

本项目当前阶段最重要的不是做复杂生态，而是把下面三件事持续做好：

1. 最小但稳定的运行时
2. 最短且清晰的导入体验
3. 可复制、可扩展、风格统一的组件开发模式

如果一个实现方案让用户导入更长、让组件更难复制、让样式更难维护，那它通常不是当前阶段的优先方案。
