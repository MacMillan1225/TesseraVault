# Obsidian Dataview 组件运行时规范（非插件版）

## 1. 目标

实现一个运行时系统，用于在 Obsidian 的 DataviewJS 环境中提供一组可复用的 JS 模块与 UI 组件。

该系统必须满足：

1. **不依赖 Obsidian 插件开发模式**
2. **只暴露一个全局命名空间**
3. **支持模块注册和模块加载**
4. **模块作者可以按接近 CommonJS 的方式写代码**
5. **用户导入模块时尽可能短、尽可能方便**
6. **适合作为 Dataview 模板组件库的基础**
7. **优先考虑核心文件，不扩展复杂工程化**
8. **支持结构化默认配置系统（fallback + `config.json` + runtime options）**

---

## 2. 总体设计原则

### 2.1 单一全局命名空间
只允许挂一个全局对象：

```js
globalThis.Tessera
```

禁止散乱挂载多个全局变量，如：

- `window.fileUtils`
- `window.cssUtils`
- `window.progress`
- `window.card`

所有能力都必须收敛到 `Tessera` 命名空间之下。

---

### 2.2 最短导入体验优先
用户最终使用时，应该尽可能接近下面这种体验：

```js
const { progress, card } = Tessera.use("components");
```

或者：

```js
const progress = Tessera.use("progress");
```

或者：

```js
const { heatmap } = Tessera.use("@ui");
```

也就是说：

- 用户不需要关心底层文件路径
- 用户不需要手写很长的模块路径
- 用户应尽量通过“短名”或“聚合入口”导入模块

---

### 2.3 模块作者保持 CommonJS 风格
模块定义应尽可能接近 CommonJS：

```js
Tessera.define("core/file", function (require, module, exports) {
  function read(path) {}
  module.exports = { read };
});
```

支持：

- `module.exports = ...`
- `exports.xxx = ...`
- `require("./relative")`

但这里的 `require` 是 Tessera 自己实现的运行时接口，不是 Node 原生的 `require`。

---

### 2.4 先注册、后使用
运行时采取：

- 先执行核心 bootstrap
- 再执行各模块文件完成注册
- 最后在 DataviewJS 中通过 `Tessera.use()` 或 `Tessera.require()` 使用

这意味着：

- 核心 `require()` 可以保持同步
- 模块体验更像 CommonJS
- 不处理复杂的异步按需加载器

---

## 3. 核心文件范围

当前只需要考虑最小核心文件，不需要写完整组件库生态。

最小核心应包含：

1. `tessera.bootstrap.js`
   - 运行时核心
   - 命名空间初始化
   - 模块注册与加载逻辑
   - 别名/短名支持

2. `index.js`
   - 聚合入口
   - 暴露高频模块集合

3. 若干核心模块示例
   - `components/progress`
   - `components/card`
   - `components/heatmap`
   - 可选 `core/dom`
   - 可选 `core/css`
   - 可选 `core/config`

重点不是组件实现细节，而是**模块系统与导入体验设计**。

同时需要注意：组件默认配置也应当被纳入统一设计，而不是散落在各组件内部。

## 4. 对外 API 规范

全局对象结构建议如下：

```js
globalThis.Tessera = {
  version,
  define,
  register,
  require,
  use,
  resolve,
  alias,
  has,
  modules,
  cache
}
```

---

## 5. 核心 API 说明

### 5.1 `Tessera.define(id, factory)`
用于定义模块。推荐首选接口。

#### 签名
```js
Tessera.define(moduleId, function (require, module, exports) {
  // ...
});
```

#### 示例
```js
Tessera.define("components/progress", function (require, module, exports) {
  function progress(options = {}) {
    const el = document.createElement("div");
    el.textContent = `${Math.round((options.value || 0) * 100)}%`;
    return el;
  }

  module.exports = progress;
  module.exports.progress = progress;
});
```

---

### 5.2 `Tessera.register(id, factory)`
这是更底层接口。若已有 `define`，则 `register` 主要供内部保留。

可以实现为：

```js
function define(id, factory) {
  register(id, ({ require, module, exports }) => {
    factory(require, module, exports);
  });
}
```

---

### 5.3 `Tessera.require(specifier, from?)`
同步加载模块。  
支持：

- 绝对模块 ID：`components/progress`
- 相对路径：`./file`
- 别名：`progress`

#### 示例
```js
const progress = Tessera.require("components/progress");
```

或者：

```js
const file = require("./file");
```

---

### 5.4 `Tessera.use(name)`
这是给最终用户使用的**高层快捷导入接口**，必须重点设计好。

`use()` 的目标是：

- 比 `require()` 更短
- 更面向用户
- 支持短名与聚合模块
- 让用户尽量不用记底层路径

#### 推荐行为
```js
Tessera.use("progress")
```
等价于：
```js
Tessera.require("components/progress")
```

```js
Tessera.use("components")
```
返回：
```js
{
  progress,
  card,
  heatmap
}
```

```js
Tessera.use("@ui")
```
也可映射到组件集合入口。

---

### 5.5 `Tessera.alias(map)`
用于注册短名映射，提升导入体验。

#### 示例
```js
Tessera.alias({
  progress: "components/progress",
  card: "components/card",
  heatmap: "components/heatmap",
  components: "index",
  "@ui": "index"
});
```

---

### 5.6 `Tessera.resolve(specifier, from?)`
处理模块路径解析。

要求支持：

1. 绝对 ID 原样返回  
   例如：
   - `components/progress`
   - `core/file`

2. 相对路径解析  
   例如在 `components/card` 中：
   ```js
   require("./shared")
   ```
   应解析到：
   ```js
   components/shared
   ```

3. 别名优先展开  
   如：
   ```js
   Tessera.require("progress")
   ```
   自动解析为：
   ```js
   components/progress
   ```

---

### 5.7 `Tessera.has(id)`
判断模块是否已注册。

#### 示例
```js
if (Tessera.has("components/progress")) {
  ...
}
```

---

## 6. 用户导入体验要求

这是本规范最重要的部分。

### 6.1 用户导入必须尽量短

优先支持以下形式：

#### 单模块导入
```js
const progress = Tessera.use("progress");
```

#### 多模块导入
```js
const { progress, card } = Tessera.use("components");
```

#### 聚合入口导入
```js
const ui = Tessera.use("@ui");
ui.progress(...)
ui.card(...)
```

不推荐用户写：

```js
const progress = Tessera.require("components/progress");
```

虽然要支持，但这属于底层方式。

---

### 6.2 推荐默认别名
系统应默认支持以下短名：

```js
progress  -> components/progress
card      -> components/card
heatmap   -> components/heatmap
components -> index
@ui       -> index
```

目标是让用户在大多数情况下只记住短名。

---

### 6.3 聚合入口必须存在
必须提供一个总入口模块，例如 `index`，统一导出最常用组件：

```js
Tessera.define("index", function (require, module, exports) {
  const progress = require("components/progress");
  const card = require("components/card");
  const heatmap = require("components/heatmap");

  module.exports = {
    progress: progress.progress || progress,
    card: card.card || card,
    heatmap: heatmap.heatmap || heatmap
  };
});
```

这样用户可以：

```js
const { progress, heatmap } = Tessera.use("components");
```

---

## 7. 模块定义规范

### 7.0 配置模块补充

除了模块注册系统本身，运行时还建议提供一个统一配置模块：

- `core/file` 负责读取 vault 文件
- `core/config` 负责读取 `config.json`、缓存结果、合并 fallback 与运行时参数

这能让组件默认值具备以下特性：

1. 有一份 JS 内置保底默认值
2. 有一份可由用户修改的 JSON 默认配置
3. 最终调用时仍然允许局部覆盖

建议最终配置模型为：

```text
fallback defaults
  + config.json defaults
  + runtime options
  = resolved config
```

---

### 7.1 模块 ID 命名规则
统一使用 `/` 分隔的命名方式：

- `core/file`
- `core/css`
- `components/progress`
- `components/card`

不要使用：

- 反斜杠
- 文件扩展名
- 随机大小写混合命名

---

### 7.2 推荐目录层级
逻辑层级建议：

- `core/*`：基础能力
- `components/*`：现成组件
- `index`：高频聚合入口

当前无需实现复杂层级。

如果组件存在稳定默认配置，则组件目录推荐同时包含：

- `index.js`
- `style.css`
- `config.json`
- `README.md`
- `demo.md`

---

### 7.3 导出风格建议

#### 单函数模块
推荐：
```js
module.exports = progress;
module.exports.progress = progress;
```

这样既支持：

```js
const progress = require("components/progress");
```

也支持聚合时命名读取：

```js
progress.progress || progress
```

#### 对象模块
推荐：
```js
module.exports = {
  read,
  write,
  normalize
};
```

## 8. bootstrap 文件必须实现的能力

`tessera.bootstrap.js` 必须完成以下事情：

### 8.1 初始化全局命名空间
如果 `globalThis.Tessera` 已存在，则直接复用或跳过初始化，避免重复覆盖。

---

### 8.2 保存模块注册表
使用一个映射结构保存模块定义：

```js
const modules = new Map();
```

---

### 8.3 保存模块缓存
用于保证模块只执行一次：

```js
const cache = new Map();
```

---

### 8.4 实现模块定义接口
支持：

- `define(id, factory)`
- `register(id, wrappedFactory)`

---

### 8.5 实现相对路径解析
要求支持：

- `./x`
- `../x`

并正确基于 `from` 模块 ID 计算。

---

### 8.6 实现别名映射
内部需要一个别名表，例如：

```js
const aliases = new Map();
```

支持：

- 注册别名
- 在 `resolve()` 时优先展开

---

### 8.7 实现 `use()`
`use()` 是高优先级用户接口。

建议逻辑：

1. 先按 alias 解析
2. 再调用 `require()`
3. 返回最终导出

---

## 9. 使用方式规范

### 9.1 初始化阶段
用户或模板入口需要先执行 bootstrap 与模块注册文件。

例如：

```js
await dv.view("Scripts/tessera.bootstrap");
await dv.view("Scripts/components.progress");
await dv.view("Scripts/components.card");
await dv.view("Scripts/components.heatmap");
await dv.view("Scripts/index");
```

说明：
- 文件名映射可由实际存储习惯决定
- 核心要求是这些脚本执行后，模块已完成注册

---

### 9.2 使用阶段
推荐用户在 DataviewJS 里这样写：

#### 方式 A：单个模块
```js
const progress = Tessera.use("progress");
const el = progress({ value: 0.6 });
dv.container.appendChild(el);
```

#### 方式 B：从组件集合中解构
```js
const { progress, card } = Tessera.use("components");
```

#### 方式 C：使用聚合命名空间
```js
const ui = Tessera.use("@ui");
const el = ui.heatmap({ data });
```

### 9.3 如果组件依赖 `config.json`

如果组件把 `config.json` 作为“首次渲染就必须生效”的默认配置来源，则推荐显式先等待：

```js
const card = Tessera.use("card");
await card.loadConfig();
dv.container.appendChild(card({ value: 12 }));
```

原因是：

- 组件函数本身通常是同步返回 DOM
- 但 `config.json` 读取是异步的
- 如果只是在组件内部触发异步加载而不等待，则第一次渲染可能只能使用 fallback

因此应明确区分两种模式：

1. 只需要 fallback 即可直接渲染
2. 需要 JSON 默认值首次就生效时，先 `await component.loadConfig()`

## 10. AI 实现时的重点要求

给 AI 的明确约束如下：

### 10.1 不要过度工程化
不要引入：

- 打包器前提
- npm 环境依赖
- TypeScript 构建前置
- 插件框架
- 异步模块加载器
- class-heavy 设计

目标是：
- 简单
- 可读
- 可复制
- 可直接在 Obsidian Dataview 环境中使用

配置系统也遵循同一原则：

- 不做复杂 schema 校验器
- 不做插件设置页绑定
- 不做远程配置中心
- 只解决 JSON 默认配置加载、缓存、合并这几个核心问题

---

### 10.2 优先保证导入体验
实现时优先保证下面三种调用都顺畅：

```js
Tessera.use("progress")
Tessera.use("components")
Tessera.use("@ui")
```

其中：

- `"progress"` 返回单个模块
- `"components"` 返回组件集合对象
- `"@ui"` 返回组件集合对象

---

### 10.3 `require()` 必须支持相对引用
例如：

```js
Tessera.define("components/card", function (require, module, exports) {
  const dom = require("../core/dom");
});
```

必须能正确解析。

但要注意：

- `require("./file")` 的相对路径解析属于 Tessera 模块系统
- `config.createScope({ path })` 的 `path` 属于 vault 文件路径

因此当前配置文件路径推荐写完整 vault 路径，例如：

```js
path: "TesseraScript/components/card/config.json"
```

而不是：

```js
path: "./config.json"
```

---

### 10.4 模块缓存必须生效
同一个模块只执行一次，后续 `require()` 返回缓存结果。

---

### 10.5 报错信息要清晰
如果模块不存在，应抛出类似错误：

```js
[Tessera] Module not found: components/xxx
```

如果循环依赖或执行失败，也应给出明确报错信息。

## 13. 不需要实现的内容

当前版本明确不需要：

- 插件设置页
- npm 包发布
- ESModule import/export 兼容
- 动态扫描 vault 自动注册
- 复杂主题系统
- 远程加载模块
- 热更新系统
- 完整组件生态

只要把**最小运行时 + 最短导入体验 + CommonJS 风格模块定义**做好即可。

配置部分同样保持克制：

- 不要求实时双向同步 JSON
- 不要求配置热更新自动重渲染
- 不要求跨组件全局配置中心
- 只要求单组件配置文件可读、可合并、可缓存
